# Rialto Agent Context

This file captures the shared language for `rialto-agent`. Keep it concise and update it when domain terms become stable.

## Product Shape

Rialto Agent is an agentic system for construction procurement. V1 is organized into three product modules: Requesting Quotes, Vendor Response Intake, and Quote Comparison. It helps contractor estimators request material quotes from vendors, collect vendor responses, and compare quotes in spreadsheet-like workflows; it stops at comparison rather than award, purchase order, or order tracking workflows.

## Domain Language

| Term | Meaning |
| --- | --- |
| Rialto Agent | The customer-facing agentic construction procurement system being built in this repository. In v1, Rialto sends quote requests, captures vendor responses, extracts response data, and automates the Excel-like quote comparison workflow, but stops at comparison rather than award or purchasing handoff. |
| Construction procurement | The contractor workflow of requesting, receiving, and comparing vendor quotes for materials needed on a job. |
| Product Module | A durable v1 product slice with its own domain interface, implementation, tests, and documentation. The active product modules are Requesting Quotes, Vendor Response Intake, and Quote Comparison. |
| Estimator | The primary user of Rialto Agent: a contractor-side user who requests material quotes from vendors and compares returned quotes. |
| Contractor Organization | The customer/account boundary for Rialto Agent. It owns users, vendor lists, quote history, templates, and procurement data. |
| Vendor | A supplier that receives material quote requests and returns pricing or availability information. |
| Vendor Contact | A person, role inbox, or email address at a vendor that can receive quote requests or provide quote responses. |
| Preferred Vendor Contact | The vendor contact Rialto suggests for a quote request based on trade fit, past responsiveness, role, and estimator preference. |
| Suppressed Vendor Contact | A vendor contact that should not receive quote requests, usually because the email is stale, the person left, or the contractor organization marked the contact as do-not-contact. |
| Inactive Vendor | A vendor that a contractor organization no longer wants to include in quote requests unless an estimator deliberately overrides the warning. |
| Vendor Specialty | A vendor's explicit or learned fit for material categories, trades, regions, or project types, used as contextual evidence rather than a single rigid classification. |
| Vendor Performance Evidence | Objective quote history signals such as response rate, response time, quote completeness, and past selection outcomes that help estimators choose vendors without turning those observations into value judgments. |
| Historical Price Benchmark | A past vendor quote used as dated context for comparison or anomaly detection, not as current pricing unless the vendor explicitly re-quotes it. |
| Vendor Recommendation | An explainable, non-binding Rialto suggestion of vendors or vendor contacts for a quote request, based on specialties, quote history, response behavior, region, project context, and estimator preferences. The estimator must approve vendors before Rialto includes them in outbound drafts. |
| Vendor Directory | A contractor organization's shared reusable list of vendor companies and vendor contacts, built from estimator-entered recipients, past quote requests, quote responses, and explicit edits. Exact email matches can normalize vendor contacts, but trust-affecting changes such as merging, deleting, or organization-wide vendor classifications require elevated permission or review. |
| Unconfirmed Vendor Contact | A vendor contact typed or pasted into a quote request before Rialto has enough evidence to treat it as part of the reusable vendor directory. Confirmation evidence includes explicit estimator confirmation, a successful vendor email send, or a vendor quote response. |
| Quote Request | The central procurement object in Rialto: an estimator's request for vendor pricing on a set of materials. It includes project information and a request expiration date, and may include a bill of materials, vendor recipients, outbound email drafts, returned vendor quotes, imported files, and a quote comparison. Rialto creates a quote request even when the estimator starts by importing an external comparison file. |
| Requesting Quotes | The product module where an estimator prepares a Quote Request, defines the Bill of Materials, chooses vendor recipients, reviews outbound vendor email drafts, and explicitly sends quote requests. |
| Quote Request Workflow | The Rialto workflow where an estimator prepares a quote request and drafts outbound emails to multiple vendors. The default starting path is uploading a file such as CSV, PDF, or Excel; manual material entry is also supported. |
| Bill of Materials | The list of materials that an estimator wants vendors to quote. It may come from an uploaded file or manual entry. Red-highlighted shared bill of materials fields that affect vendor scope block the quote request send until reviewed; vendor-specific unresolved fields block only the affected vendor draft. |
| Vendor Alternate | A vendor-proposed substitution, alternate item, exclusion, or scope variation linked to a requested line item but kept separate from the main comparison until the estimator decides how to compare it. Rialto should only mark a vendor response as an alternate/substitution when the vendor explicitly submits it as one, such as using the substitution action in the magic RFQ form. Different wording, SKU text, imported notes, or imported quote file language like "alternate manufacturer" should remain source notes unless the explicit alternate flag is present. An estimator may select a vendor alternate for a line item, and that line then counts as resolved, but Rialto must preserve that the selection differs from the original requested item. A selected alternate does not count toward complete-comparable lowest quote calculations unless the estimator explicitly accepts the scope change for comparison. |
| Substitution Spec Verdict | The AI-assisted determination for an explicit Vendor Alternate after comparing the vendor's substituted material facts against the Project Spec Package. The verdict is `up_to_spec`, `not_up_to_spec`, or `needs_review`, and it must include only the relevant cited spec lines used for the decision rather than the full Project Spec Package. |
| Scope Acceptance | A line-specific estimator decision that a vendor alternate's changed scope is acceptable for comparison against the requested line item. Scope acceptance is distinct from vendor selection: vendor selection records what the estimator intends to use, while scope acceptance controls whether Rialto may treat the alternate's price as comparable in lowest complete quote calculations. After scope acceptance, package summaries should still show that the line was resolved using accepted alternate scope. |
| Quantity Mismatch | A vendor quote response where the quoted quantity or unit differs from the requested quantity or unit. Rialto should preserve the vendor's quoted quantity and unit, may show simple comparison math or packaging-unit interpretations when the unit and pricing basis are clear, and should keep the mismatch as a comparison issue for estimator review before deciding whether it is corrected data, a normalized quote, accepted vendor quantity, or a vendor alternate with changed scope. A normalized quote is an estimator-approved working comparison value brought back to the requested quantity; the vendor's original quantity and value remain provenance. If normalization scales a vendor quote up to a larger requested quantity, Rialto should caveat that availability at the requested quantity may still need confirmation. If normalization scales a vendor quote down to a smaller requested quantity, Rialto should caveat that minimum order, package size, or actual buy quantity may still need confirmation. Unit changes with unclear conversions are higher-risk and should require review before normalization. Procurement memory may suggest a prior quantity or unit normalization, but in v1 Rialto should not auto-apply it. Until resolved, affected line values and roll-up totals may be shown but should not be treated as fully comparable, and export should warn rather than block. When an estimator resolves the mismatch as satisfying the requested need, Rialto should allow the line to count as complete for comparison, remove the unresolved red review highlight, and preserve the original quantity difference and any caveats in provenance, audit history, or cell details. |
| No-Bid Line Item | A requested line item with no usable vendor quote value, whether the vendor explicitly declines, leaves it blank, does not respond, or cannot supply it. Explicit no-bid lines and blank or missing quote values both make the vendor quote incomplete for comparison, but Rialto should preserve whether the source was an explicit decline or an ambiguous missing value. A vendor response with no-bid lines can still be useful and valid as a partial quote response, but should not compete as that vendor's complete quote. No-bid line resolution should come from explicit selection state such as selecting another vendor for the line, marking the line out of scope, deferring the decision, or marking no award; resolving the package through line-level selections does not make the original partial vendor quote complete. Rialto should not invent a price to make a vendor quote appear complete, and must account for no-bid line items when calculating quote totals so partial quote responses do not create misleading totals. |
| Partial Quote Total | A vendor total calculated from only the line items with usable quote values when one or more requested line items are no-bid or missing. Rialto may display a partial quote total, include it in ordinary table sorting when clearly labeled, but must show the missing/no-bid line items or count and avoid presenting the vendor as fully comparable with complete quotes. Rialto's lowest complete comparable quote logic must not crown a partial quote total as the lowest complete quote. |
| Quote Request Attachment | A file sent with a vendor email draft, usually shared across vendors but optionally overridden per vendor when the estimator needs vendor-specific context. |
| Project Spec Package | A trade-scoped subset of uploaded project specification manuals created once after project spec indexing, based on the contractor organization's configured trade. It keeps only the spec chunks likely to govern that contractor's work plus common product requirement, submittal, approved-equal, and substitution-control language. Quote-time spec checks should prefer this package over retrieving from the entire project manual so 1000+ page manuals do not get repeatedly re-parsed or over-broadened. |
| Vendor Email Draft | A per-vendor email draft prepared visibly by Rialto in an in-app email composer for estimator review, manual editing, and sending. Vendors should receive separate drafts rather than a shared email thread, even when the request content is mostly identical. A draft may include multiple contacts for the same vendor, but Rialto treats that as one vendor invitation. In v1, the estimator reviews drafts in a batch flow and can send all approved drafts through their authenticated Google or Microsoft 365 mailbox with an explicit final send action. |
| Vendor Quote Form | A no-login form opened from a magic link in a vendor email. Vendors use it to submit and edit quote details back into Rialto until the quote request expiration date. The form shows structured line items when available and supports structured line-item entry, alternates, substitutions, exclusions, notes, quote file uploads, and email replies with vendors' usual quote files. |
| Vendor Invitation | A vendor company's invitation to respond to a specific quote request, delivered to one or more vendor contacts. |
| Magic Link | The hard-to-guess, revocable, per-vendor-invitation, no-login link in a vendor email draft that opens the vendor quote form for the associated quote request. Anyone with the link can submit for that vendor invitation in v1. The link expires on the quote request expiration date unless the estimator extends or regenerates access. |
| Vendor Quote Response | A vendor's submitted pricing, availability, notes, exclusions, or attachments for a quote request, whether received through the vendor quote form or by email reply with body text and attachments. Multiple contacts at the same vendor still roll up to the same vendor's response history for the quote request. When a vendor edits a response before expiration, the latest submission is current while prior submissions remain in response history. |
| Vendor Response Intake | The product module where Rialto receives, extracts, associates, reconciles, and merges Vendor Quote Responses into structured quote data while preserving source artifacts and review issues. |
| Quote Response Extraction | Rialto's reading of vendor quote response sources to extract pricing, availability, notes, exclusions, attachments, and other quote details into structured procurement data suitable for quote comparison. High-confidence fields can merge automatically, but uncertain price, quantity, scope, alternate, exclusion, or vendor identity data requires estimator review. |
| Vendor Quote Draft | The editable structured draft produced by Vendor Response Intake from a Vendor Response Source before it is submitted or merged. Vendor Magic Link uploads, inline email replies, and email attachments should all use this draft shape so source parsing, RFQ line matching, product-detail warnings, and provenance live behind one module interface. |
| Vendor Response Source | The original form submission, email body, PDF, spreadsheet, or other attachment from which Rialto extracts vendor quote response data. Rialto preserves vendor response sources as the audit trail for extracted quote data. |
| Quote Value Provenance | The trace from an extracted quote value in a comparison sheet back to the vendor response source and source location that produced it. |
| Extraction Feedback | An estimator correction to extracted quote data that updates the current comparison and gives Rialto evidence for improving future extraction from similar sources, vendors, or file patterns. |
| Quote Response Association | Rialto's confidence that a vendor email reply or submission belongs to a specific quote request, based on signals such as email thread, sent quote email metadata, magic links, subject patterns, and known vendor contacts. |
| Possible Vendor Quote Response | A vendor email or submission that may belong to a quote request but needs estimator confirmation before Rialto treats it as a vendor quote response or merges it into procurement history. |
| Vendor Response Reconciliation | Estimator review of conflicting or overlapping quote responses from the same vendor, assisted by Rialto comparisons and source artifacts. |
| Quote Response Merge | The process of adding extracted vendor quote response data into the comparison sheet. Once a quote request is configured, Rialto automatically extracts and merges vendor responses with visible progress, provenance, and review highlights when vendor responses arrive or when a vendor submits a newer response. Estimator review is only needed when Rialto cannot confidently interpret or place the response. |
| Vendor Response Workbook | The spreadsheet handoff produced by the Quote Request workflow and consumed by the Quote Comparison workflow. It contains the requested bill of materials, vendor response columns, source/provenance metadata, and review issues in a format that can be opened as an Excel workbook or imported from a client-provided workbook. |
| Deterministic Workbook Tools | Backend spreadsheet tools that ingest workbook-shaped data, extract sheet/table schema, read ranges, search/query rows, analyze RFQ quote columns, create risk-scored JSON patch previews, apply approved patches with verification, append audit events, and rollback by workbook version. The OpenAI Agents SDK orchestrator may call these tools, but the LLM does not directly mutate workbook files or generate free-form spreadsheet code. |
| Workbook Handoff | The modular seam between Quote Request and Quote Comparison. Quote Request may produce a Vendor Response Workbook after collecting responses, while Quote Comparison may also start directly from an externally provided Vendor Response Workbook when a client already has its own request workflow. |
| Quote Comparison | The spreadsheet-like workflow where an estimator compares returned vendor quotes and chooses how to proceed. A quote comparison can start from a Rialto-facilitated quote request or from an externally imported file. When complete and partial vendor quotes are mixed, Rialto may identify the lowest complete comparable quote and may separately call out lower partial totals with missing lines, but must not present a partial quote total as the lowest or best complete quote. |
| Quote Comparison Agent Tools | Backend-callable Rialto Agent tools owned by the Quote Comparison product module. They expose visible Comparison Sheet operations such as editing cells, deleting or hiding rows and columns, inserting rows and columns, renaming columns, Excel-like sorting, filtering, derived columns, and highlights. The browser assistant adapts tool results from one estimator request into a single previewable Comparison Sheet patch proposal. |
| User Context | The full readable app and database state available to Rialto Agent for the authenticated user, including pages, projects, quote requests, comparison sheets, vendor data, files, and procurement history. User Context is broader than Procurement Memory; it is the operating context the intelligence core may inspect before choosing visible product tools. |
| Comparison Sheet | The live mutable working workbook for a quote comparison. Rialto creates the initial comparison sheet from the bill of materials before vendor responses arrive so responses have a structure to merge into. Estimators and Rialto's agent edit the same server-persisted sheet state, and the currently visible workbook values and structure are the source of truth for export. Deleting rows or columns removes them from the working workbook and export, but does not destroy the underlying quote request, vendor response, source file, provenance, or history records. Rialto's agent interprets procurement meaning such as vendors, material line items, prices, lead times, notes, alternates, and missing values. Corrected values become the working values, while original extracted values and correction events remain available in sheet history. |
| Comparison Sheet Snapshot | A hydrated readable view of a Comparison Sheet that combines persistent procurement data with comparison view state into the rows, columns, cells, highlights, hidden state, manual structure, overrides, and provenance references the estimator currently sees; it is the agent's current sheet state for Quote Comparison reasoning. |
| Vendor Selection | An estimator's chosen vendor for a material line item in a quote comparison. Rialto allows selecting a vendor even when that vendor's quote is partial or before every missing line is resolved, but the selection must visibly caveat that missing or no-bid line items remain unresolved. Line-level selections are the source of truth; package-level and whole-quote selections are roll-up summaries. A package with a selected partial vendor remains unresolved until every missing or no-bid line item has its own explicit selection state. In v1, vendor selection is comparison sheet state only and does not trigger award, notification, decline, or purchasing handoff workflows. Past selections become contextual procurement memory for future vendor recommendations. |
| Selected Package Total | A package roll-up total calculated from the estimator's line-level priced selections, potentially across multiple vendors. It uses estimator-approved working values, including accepted alternates and resolved normalized quantity values, while preserving provenance and caveats. When some in-scope lines are deferred or no-award, Rialto should label the amount as a selected priced total rather than a full package cost. It is distinct from any single vendor's quote total and should separately show no-award lines, deferred lines, out-of-scope lines, accepted alternates, quantity mismatch caveats, and other unresolved risks. |
| Selection State | The decision state for a material line item or package, such as selected vendor, no award, deferred decision, or out of scope. Marking a line out of scope removes it from the required comparison scope for all vendors in that comparison scope rather than resolving only one vendor's missing value; complete totals and lowest complete comparable quote calculations should then use the remaining in-scope lines while preserving the original requested line in history. No award and deferred decision can close review on a line for the current comparison workflow, but unlike out-of-scope they do not remove the line from the required comparison set; package summaries and exports should preserve that the estimator intentionally chose not to award or decide that in-scope line. A package can count as resolved when every in-scope line has an explicit selection state, even if some lines have no vendor price; package summaries should distinguish priced selections from no-award, deferred, and out-of-scope lines. |
| Selection Follow-up | A post-comparison action outside the v1 product boundary, such as notifying selected vendors, declining unselected vendors, awarding work, or preparing purchasing handoff materials. |
| Purchasing Handoff | A traceable package of selected vendors, selected prices, quantities, alternates, exclusions, lead times, quote validity dates, vendor contacts, source quote files or emails, estimator notes, and unresolved risks or assumptions for downstream purchasing. Purchasing handoff is outside the v1 product boundary. |
| External Quote Import | One or more PDF, CSV, Excel, XML, text, or other quote files brought into Rialto from an outside workflow so Rialto can populate a quote comparison. In v1, every uploaded quote first attempts the normal deterministic extraction and comparison importer; if extraction or deterministic quote-row parsing fails, the import module privately falls back to GPT-5.5 normalization and then re-enters the same deterministic importer. Imported quote notes and clarifications are preserved as notes only and must not set the alternate/substitution highlight. Rialto preserves the imported files as source artifacts and works on an editable copy inside the comparison sheet. During the current import hardening phase, the comparison page may show whether a file used normal import or GPT-5.5 fallback and include the fallback exception reason for debugging. |
| Comparison Formula Dependency | A visible Comparison Sheet relationship where changing a vendor Unit Price should recompute that vendor's Total Price for the same line from the requested quantity. The dependency is represented through the Comparison Sheet edit/override path so imports, manual edits, agent proposals, history, and export stay on the same workbook state rather than mutating hidden bid source data. |
| Workbook Version | An automatic or explicitly named saved state of the comparison sheet that lets Rialto preserve history and support recovery from imports, extractions, merges, estimator corrections, user edits, and agent edits. Rialto persists the latest estimator-visible workbook state separately for fast loading and appends immutable version records with source, actor, summary, parent version, and optional agent proposal provenance. Restoring an older workbook version creates a new version instead of deleting later history and does not erase vendor responses or source artifacts from quote request history. |
| Comparison Timeline | The estimator-facing history of meaningful comparison milestones such as imports, extractions, merges, corrections, review highlight acknowledgments, and agent edits, with deeper workbook version history available for audit or recovery. |
| Comparison Export | A clean export of the current Comparison Sheet state and selected package totals by default, including estimator and agent mutations to visible workbook values, rows, columns, labels, ordering, and formulas, without Rialto review highlights or flags. Rialto warns but does not block export when unresolved review highlights remain, and offers a separate audit export option that includes vendor quote totals, partial labels, provenance, source links, extraction confidence, corrections, selection decisions, and comparison timeline history. |
| Visible Agent Edit | A spreadsheet or webapp change performed by Rialto's agent in front of the estimator inside the webapp, so the estimator can see what the agent is doing. In v1, Rialto's agent focuses on guided editing and automation of the Excel-like quote comparison workflow. Agent actions execute through internal product tools and are visualized in the UI; large mechanical edits may be applied in batches while still showing progress and highlighting affected ranges afterward. Material agent changes to values, formulas, formatting, row or column structure, sorting, filtering, grouping, or notes create orange review highlights until acknowledged by the estimator by range, batch, or all current highlights. Direct estimator edits are saved in workbook history but do not create orange highlights. |
| Comparison Patch Proposal | The complete previewable set of Comparison Sheet changes produced from one estimator request, overlaid on the live sheet state and approved or discarded as a single unit. Cells and ranges affected by a proposal should be visually previewed in yellow so the estimator can quickly see what Rialto is about to touch. |
| Fast Comparison Command | A deterministic, low-latency Visible Agent Edit path for narrow Quote Comparison instructions where the target cells or highlights are mechanically clear from the current Comparison Sheet Snapshot, such as setting all alternate cells to a value or highlighting the fastest lead time for a named line item. Fast comparison commands still return a previewable Comparison Patch Proposal shape and do not replace the Product Agent Runtime for broad procurement reasoning or ambiguous requests. |
| Agent Work Mode | The Product Agent Runtime's categorization of an estimator request before it chooses tools: answer-only for requests that should not change the sheet, fast sheet adjustment for narrow mechanically clear edits that should feel almost instant, and deep sheet adjustment for larger or ambiguous work that needs broader reasoning across tools before returning a plan, clarification, answer, or patch proposal. |
| Deep Sheet Adjustment | An Agent Work Mode for broad, multi-step, or materially ambiguous Quote Comparison work. Rialto should classify the request, inspect the visible sheet and relevant user context, analyze findings and assumptions, ask one concise clarification when needed, and only then prepare a single yellow-previewed Comparison Patch Proposal. |
| Product Agent Runtime | The OpenAI-backed Rialto Agent runtime that interprets estimator requests, inspects User Context, calls product-module tools, and returns visible plans, clarification questions, or proposal outputs. It requires `OPENAI_API_KEY` and should not be duplicated by a deterministic fallback planner. |
| Agent Debug Trace | An optionally visible webapp trace of a Rialto Agent turn showing the agent's response state, plan, tool calls, tool results, patch fragments, aggregated proposal, warnings, and errors. It is for development and support inspection, not the normal estimator approval surface. |
| Bulk Comparison Edit | A Visible Agent Edit that applies one instruction across many Comparison Sheet cells or rows, such as increasing every Unit Price and recomputing dependent Total Price cells. Bulk comparison edits should be represented as Quote Comparison Agent Tools rather than many unrelated one-off cell edits, then expanded into a Comparison Patch Proposal for estimator review. |
| Vendor Update Highlight | A blue comparison sheet review highlight showing cells changed because Rialto updated the comparison from a newer vendor quote response; it remains until acknowledged by the estimator by range, update batch, or all current highlights. |
| Extraction Review Highlight | A red comparison sheet review highlight showing extracted values that need estimator review before they should be trusted, such as uncertain price, quantity, scope, alternate, exclusion, or vendor identity data. Acknowledging red means the estimator reviewed the uncertainty by accepting, correcting, or intentionally leaving the value as-is. |
| Review Highlight Stack | Layered unacknowledged comparison sheet highlights for the same cell or range, where the top visible color reflects the latest unacknowledged review event and acknowledging one layer reveals any unacknowledged layer beneath it. Each layer has an explanation, timestamp, source or action, and acknowledgment status. Export warnings count all unresolved layers, including hidden layers. |
| Proposal Highlight | A yellow visible highlight included in a Comparison Patch Proposal. When an estimator asks Rialto to highlight cells or ranges without naming a specific color, Rialto should propose yellow highlights by default. |
| Agent Plan | A visible plan shown by Rialto's agent for multi-step, broad, risky, structural, formula-changing, bulk, import, or external-consequence work before mutating the live Comparison Sheet. Tiny reversible edits may be executed without showing a plan first. After approval, planned steps execute against the same live workbook state the estimator sees and exports. |
| Agent Planning Analysis | A read-only Product Agent Runtime tool result that classifies an estimator request as a simple narrow command or broader work that needs planning, summarizes visible sheet risk signals, and recommends the next Quote Comparison tool families before the agent answers, asks for clarification, or prepares a patch proposal. |
| Procurement Memory | Rialto's saved practical procurement context for an estimator or contractor organization, including the shared vendor directory, organization-visible quote history, preferred email wording, signatures, estimator preferences, material naming and unit normalization, past quote requests, source files, comparison sheets, and selected vendors. Sensitive source artifacts may still have narrower access controls. |
| Skill | A reusable Codex instruction bundle installed outside this repo, usually under `~/.codex/skills`. Skills augment the agent but are not product code. |
| Agent docs | Repo-local guidance files such as `AGENTS.md`, `CONTEXT.md`, and `docs/agents/*` that teach agents how to work on this specific product. |
| ADR | An architecture decision record under `docs/adr/` that explains a decision future agents and maintainers should preserve or consciously revisit. |
